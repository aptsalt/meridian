The Problem

Banks want GenAI over policy docs.
Regulators want accountability.

BMO's Lumi, every bank's AI roadmap — the pattern is the same: put a generative assistant over thousands of internal policy documents. But in a regulated institution, a raw LLM answer is not enough.

Without governance

  • Hallucinated policy references cited with confidence
  • No citation trail — regulators can't verify the basis of advice
  • Material financial decisions made without advisor sign-off
  • No tamper-evident record of what the AI said and when
  • PII leaks, prompt injection, model jailbreaks

What governance requires

  • Grounded — every claim sourced from retrieved documents
  • Cited — policy IDs visible in every response
  • Governed — human approval gate for material answers
  • Auditable — tamper-evident log, hash-chained
  • Secured — DPoP tokens, BFF, scoped access

Compliance bottom line: Ungoverned GenAI in banking is a regulatory exposure, not a product feature. The answer is a transparent, grounded, human-supervised AI architecture — and that's exactly what Meridian is.

Retry on weak retrieval

If grader scores context < 0.7, the agent re-queries with a rewritten question before generating — eliminating hallucinated citations.

HITL approval gate

Material answers (rate quotes, policy exceptions) pause the stream and require explicit advisor sign-off before delivery and logging.

Streaming UI

The Angular frontend subscribes to SSE, rendering tokens, tool-call badges, and citation chips as they arrive — no loading spinner.

Hash-chained audit

Every approved decision is written to an append-only log where each record includes SHA256(prev_hash + payload). Tamper is detectable.

Grounding · Citations

Every answer is grounded or refused.
No exceptions.

The agent is architecturally prevented from generating answers without retrieving supporting context first. The UI exposes this transparently — the advisor always knows why the AI said what it said.

Citation chips — always visible

AI response excerpt:
Under international payments policy, PAY-114 wire transfers above $10k require dual approval. See also API-330 for API rate limits on bulk payment endpoints.
Groundedness: 0.94 GROUNDED
When grounding fails:
⚠ Insufficient policy context retrieved for this query. I cannot provide a grounded answer. AI-601
Groundedness: 0.41 — below threshold

How grounding works

  • Query is embedded and searched against policy vector store
  • Retrieved chunks graded for relevance (0–1 score)
  • If score < 0.7, agent rewrites and retries (up to 3×)
  • LLM generates only from retrieved context, not parametric memory
  • LLM-as-judge (rag-eval-engine) evaluates final answer faithfulness
  • Citation IDs embedded structurally — UI renders them as chips

Policy AI-601 — the system's own policy for AI behavior. When the model cannot ground an answer, it returns this code and a refusal. The refusal itself is audited.

Human-in-the-loop

  • Agent classifies response materiality before delivering
  • Material answers (rate quotes, policy exceptions, credit decisions) trigger a pause state
  • Angular UI renders an approval panel — advisor reads full context, sources, and groundedness
  • One-click Approve or Reject with optional override note
  • Only after approval does the answer reach the end-user and get logged

Hash-chained audit trail

  • Append-only log — no update or delete operations
  • Each record stores SHA-256(prev_hash + timestamp + payload)
  • Chain integrity verifiable client-side — any tampering breaks the chain
  • Stores: query, retrieved docs, groundedness score, advisor decision, timestamp, approver ID
  • Meets financial institution audit trail requirements

MCP Tool Calls

The agent calls typed tools.
Live. Visible. Audited.

Model Context Protocol (MCP) tools extend the agent beyond language. Each call fires in real time and is rendered as a live badge in the Angular streaming UI.

policy_search
Semantic search across the bank's full policy corpus. Returns ranked chunks with doc IDs and relevance scores.
→ refs: [PAY-114, API-330, COMP-220]
compliance_check
Validates a proposed action against regulatory rules. Returns pass/fail with the specific rule violated or satisfied.
→ COMP-220 · status: PASS · KYC required
portfolio_scan
Reads anonymised portfolio summary for the client context. Triggers materiality classification for HITL gating.
→ exposure: HIGH · hitl_required: true
rate_lookup
Fetches current rates (FX, prime, overnight) from the rates service. Result is always cited and timestamped in the answer.
→ CAD/USD: 0.7312 · as-of: 2026-06-23T14:00Z
Live in Angular UI as the agent runs: ▶ policy_search → ▶ compliance_check → ▶ portfolio_scan → ▶ rate_lookup → streaming answer with citations

MCP over HTTP

Tools exposed as a typed MCP server, consumed by the LangGraph agent. Adding a new tool = one JSON schema, zero agent rewrites.

Angular tool stream

SSE events carry tool-call metadata. Angular renders a live activity feed — advisors see exactly what the agent is doing.

Security Architecture

Banking-grade auth. Sender-constrained tokens.
Defence in depth.

OAuth 2.0 + OIDC
  • Auth Code + PKCE
  • PAR (Pushed Auth Req)
  • Short-lived access tokens
  • Refresh token rotation
DPoP Binding
  • Client generates EC key pair
  • DPoP proof JWT per request
  • Token bound to public key
  • Stolen token = useless
BFF Pattern
  • Tokens never in browser
  • HttpOnly cookie session
  • BFF proxies API calls
  • XSS can't steal tokens
Angular Guards
  • Functional route guards
  • Scope-aware CanActivate
  • Role-based route blocks
  • Interceptor attaches proof
// dpopInterceptor (functional)
export const dpopInterceptor: HttpInterceptorFn =
(req, next) => {
const proof = dpopService.generateProof(req);
return next(req.clone({
setHeaders: { DPoP: proof }
}));
};

Why DPoP beats Bearer

A stolen Bearer token works anywhere. A DPoP-bound token requires the matching private key — making credential theft attacks impractical.

Scoped access

Route guards check token scopes (policy:read, audit:write, hitl:approve) — least-privilege enforced at the Angular layer.

AI Guardrails

Two layers of guardrails.
Guardrails make teams faster.

Guardrails aren't friction — they're the reason you can ship AI to a regulated bank at all. They compress the review cycle and build trust with compliance teams.

Dev-Time Guardrails
  • Prompt boundaries — system prompt defines scope, persona, and refusal policy; never in user-editable fields
  • Golden test patterns — CI runs a prompt regression suite; any answer drift on known queries fails the build
  • CI linters — ESLint + strict TypeScript catch unsafe string interpolation into prompts at commit time
  • Local AI reviewer — git-sentry (my own tool) runs an Ollama-backed review on every diff before push
  • Type safety — all agent I/O typed with Zod schemas; untyped inputs never reach the LLM
  • Human review gate — prompt changes require peer approval in PR regardless of CI status
Run-Time Guardrails
  • RAG grounding — model cannot answer without retrieved context; generates only from chunks, not parametric memory
  • Mandatory citations — structured output schema requires citation IDs; empty citations = refusal
  • LLM-as-judge eval — rag-eval-engine scores faithfulness, relevance, and groundedness post-generation
  • PII redaction — Presidio-based pipeline strips names, accounts, SINs before logging
  • HITL gate — material classifications (high exposure, exceptions) pause delivery for advisor approval
  • Tamper-evident log — every guardrail outcome is recorded; deviations trigger alerts

My own git-sentry tool (4-agent, Ollama local AI) enforces dev-time guardrails on this codebase. Every commit goes through it before it can push. The AI that builds the AI also reviews it.

Modern Angular · Under the Hood

Angular 21. Zoneless. Signals-first.
Ready for Angular 22.

Zoneless CD
No zone.js. Change detection driven purely by signal mutations. Zero unexpected re-renders. Measurably faster apps.
Signals + computed()
signal(), computed(), effect() replace BehaviorSubject for all derived state. Synchronous, lazy, garbage-collected.
signal input()
Component inputs are signals — readonly query = input<string>(). No ngOnChanges. Computed derivations just work.
Standalone + Lazy
All components standalone. Routes lazy-load with loadComponent. NgModule files: zero. Tree-shaking is surgical.
@if / @for / @switch
Built-in control flow blocks replace *ngIf and *ngFor. Template reads as code. No structural directive imports needed.
@defer
Heavy chart and diagram components deferred until visible. Reduces initial bundle by ~40%. Built into the template — no router tricks.
Functional Interceptors
HttpInterceptorFn — the DPoP interceptor is a plain function, not a class. Easier to test, compose, and tree-shake.
Functional Guards
CanActivateFn with injected signals. Route protection reads scope from the auth signal — no service class overhead.
OnPush Everywhere
Every component is OnPush. With signals, the framework knows exactly which component subtrees to re-check — and skips the rest.
Signal-based State
No NgRx, no BehaviorSubject store. A ChatStateService with signals and computed projections handles all async state.
Preload Strategy
Custom preload strategy warms the policy-copilot remote during idle time — so navigating to it is instant despite lazy loading.
Angular 22 Ready
No deprecated APIs. All patterns align with Angular team's stated roadmap. This codebase upgrades with one ng update.

  • Teams deploy independently — no release train, no freeze windows
  • Each remote has isolated CI/CD, own test suite, own Slack channel
  • Shared singletons (auth, design system) loaded once, version-locked
  • A/B test the copilot remote without touching payments
  • Roll back one remote without a full portal redeploy

Technical implementation
